# 检索引擎设计

## 1. SKILL 与知识层映射

### 1.1 SKILL → 知识层映射表

| SKILL | 知识层 | 数据来源 |
|-------|--------|---------|
| **SKL001** 运单追踪 | 公司层 + 订单层 | 调用 TMS Tool |
| **SKL002** 报价 | 平台层 + 公司层 | 调用 TMS Tool + 知识库 |
| **SKL003** 库存查询 | 公司层 | 调用 WMS Tool |
| **SKL004** 创建订单 | 公司层 + 订单层 | 调用 TMS/OMS Tool |
| **SKL005** 知识检索 | 平台层 + 公司层 + 用户层 | 三层知识库 |
| **SKL006** 客户信息 | 公司层 + 用户层 | 公司层 + 用户授权 |

### 1.2 Tool 调用 vs 知识库检索

```
Tool 调用（实时数据）
├── TMS Tool      → 运单追踪、创建订单
├── WMS Tool      → 库存查询
└── OMS Tool      → 订单查询

知识库检索（RAG）
├── 平台层        → 行业基础知识
├── 公司层        → 客户/产品/SOP
└── 用户层        → 个人话术/模板
```

## 2. 检索流程

### 2.1 整体流程

```
用户 query
    │
    ▼
┌─────────────────────────────────────┐
│  Step 1: SKILL 匹配                  │
│  匹配成功 → 执行 SKILL → 调用 Tool   │
│  匹配失败 → 进入知识检索流程          │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  Step 2: 知识分类路由                 │
│  按 SKILL 类型确定候选知识层          │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  Step 3: 语义补充判断                │
│  AI 语义分析是否需要扩展检索层       │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  Step 4: 权限过滤                    │
│  按用户角色过滤可访问的知识          │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  Step 5: 合并排序                    │
│  多层结果合并，按相关度排序          │
└─────────────────────────────────────┘
    │
    ▼
返回结果
```

### 2.2 分类路由规则

知识检索（SKL005）时，按分类自动路由：

| 查询关键词 | 路由到 | 说明 |
|-----------|--------|------|
| 海关/出口/进口/关税 | 平台层 | 海关政策类 |
| FedEx/DHL/UPS/顺丰 | 平台层 | 快递公司类 |
| Amazon/eBay/Shopify | 平台层 | 电商平台类 |
| FBA/海外仓/直邮 | 平台层 | 物流方案类 |
| 公司SOP/流程/操作 | 公司层 | SOP流程类 |
| 客户/联系人 | 公司层 | 客户信息类 |
| 产品/价格/航线 | 公司层 | 产品信息类 |
| 运单/订单状态 | 订单层 | 订单数据 |
| 个人话术/模板/笔记 | 用户层 | 用户个人数据 |

### 2.3 兜底策略

当分类路由未命中时，按以下顺序兜底：

```
用户层 → 公司层 → 平台层
```

## 3. 语义补充判断

### 3.1 判断逻辑

AI 根据 query 语义判断是否需要扩展检索层：

```
如果：分类路由已命中多层
    且：结果数 < 阈值
    则：尝试扩展到其他相关层

如果：分类路由未命中
    则：AI 判断意图，扩展检索层
```

### 3.2 示例

| 用户输入 | 分类路由 | 语义补充 | 最终检索层 |
|---------|---------|---------|---------|
| "发美国亚马逊怎么走" | 平台层(物流方案) | 补充查公司层(产品信息) | 平台层 + 公司层 |
| "客户张总要发货" | 公司层(客户信息) | 用户层(个人笔记) | 公司层 + 用户层 |
| "帮我查下这个运单" | 订单层 | 扩展查公司层(SOP) | 订单层 + 公司层 |

## 4. 权限过滤

### 4.1 层级权限

| 层级 | 可见条件 |
|------|---------|
| 平台层 | 全部可见 |
| 公司层 | 属于同一公司 |
| 订单层 | 属于同一公司 + 订单阶段权限 |
| 用户层 | 仅本人 + 公司授权的内容 |

### 4.2 订单阶段权限

| 角色 | 可看到阶段 |
|------|---------|
| 销售 | 待处理、执行中、已完成 |
| 客服 | 全部（含异常） |
| 仓库 | 执行中、异常 |
| 管理员 | 全部 |

---

## 5. 结果合并与排序

### 5.1 合并规则

```
多层的知识结果 → 合并去重 → 按相关度排序
```

### 5.2 排序因素

| 因素 | 权重 | 说明 |
|------|------|------|
| 文本相关性 | 40% | 向量相似度得分 |
| 权限优先级 | 30% | 用户层 > 公司层 > 平台层 |
| 时效性 | 20% | 最新更新的优先 |
| 使用频率 | 10% | 高频知识优先 |

## 6. 检索 API

### 6.1 知识检索接口

```
POST /api/v1/knowledge/search
```

**请求体：**
```json
{
  "query": "发美国亚马逊怎么走",
  "skill": "search_knowledge",
  "layers": ["platform", "company"],
  "filters": {
    "category_ids": ["PLT004"],
    "company_id": "C001"
  },
  "limit": 10
}
```

**响应体：**
```json
{
  "results": [
    {
      "id": "K001",
      "title": "美国亚马逊FBA头程方案",
      "content": "...",
      "layer": "platform",
      "category": "物流方案",
      "relevance_score": 0.95,
      "source": "RGA知识库"
    }
  ],
  "total": 15,
  "query_time_ms": 45
}
```

### 6.2 订单检索接口

```
POST /api/v1/orders/search
```

**请求体：**
```json
{
  "query": "帮我查下WB20260328001",
  "skill": "track_waybill",
  "company_id": "C001",
  "user_id": "U001",
  "stage_permission": ["pending", "processing", "completed", "exception"]
}
```
